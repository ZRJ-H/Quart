# 方法论整合：采集管道统一与 Karpathy 知识体系全量摄取

## 分析摘要

今日完成 wiki 知识库采集管道的三合一统一（B站 + YouTube + 文章），并以 Karpathy 全量教学素材（11视频+5文章）验证了端到端工作流。本文分析此过程中呈现的**三条内在主线**：管道架构的统一逻辑、Karpathy 知识体系的结构特征、以及它们共同揭示的 wiki 方法论升级方向。

---

## 主线一：采集管道三合一 — 从单点到平台

### 管道演进

之前 wiki 内容来源只有两条：B站世界周刊（被动/周期性）和 GitHub Trending（自动化但浅层）。采集依赖单一 MCP，缺乏通用的文字资料采集能力。

今日新增第三条管道后，三者构成完整矩阵：

```
来源类型       采集方式           原始存储              摄取模式
──────────    ────────────      ─────────────         ───────────
B站视频  ─→  bilibili-save.py  → raw/bilibili/       → subagent (已有33期)
YouTube  ─→  youtube-save.py   → raw/youtube/        → subagent (已验证)
文章     ─→  article-save.py   → raw/articles/       → subagent (已验证)
```

### 关键设计决策

| 决策 | 理由 |
|------|------|
| 文章不分子目录 | 用户说"堆在一起就行"，降低认知负担 |
| 独立脚本而非 MCP | 批量采集需要独立控制流，MCP 适合交互式查询 |
| 统一的 subagent 摄取模式 | 4 个 subagent 同时处理 16 项素材验证了并行摄取的可扩展性 |
| yt-dlp 而非 MCP 采集长视频 | MCP 的 youtube-transcript 对 3h+ 视频超时，yt-dlp 稳定 |

### 管道验证指标

- 11 部视频平均采集时间 ~20-60 秒（元数据 + 字幕下载）
- 5 篇文章平均采集时间 ~5-10 秒（HTTP 请求 + HTML 解析）
- 4 个并行 subagent 全部成功，零冲突，零重复文件
- 来源页命名一致性：`karpathy-{slug}.md` 统一前缀

## 主线二：Karpathy 知识体系 — 五层渐进结构

Karpathy 的素材不是零散的内容，而是一个**精心设计的五层学习架构**。将其完整摄入 wiki 后，这个结构清晰浮现：

### 第一层：概念框架 — "是什么"

| 素材 | 核心输出 |
|------|----------|
| [1hr Talk] Intro to LLMs | LLM 的基本概念、训练流程 |
| Deep Dive into LLMs like ChatGPT | 完整训练栈 + 心理学模型 + 实用建议 |
| How I use LLMs | LLM 作为操作系统，工具使用模式 |

→ Wiki 概念页：LLM即操作系统、系统1/2思维、推理模型、上下文窗口、工具使用

### 第二层：基础实现 — "怎么建"

micrograd → makemore 系列（5 部）逐步构建：

```
micrograd (反向传播引擎) 
    → makemore (字符级bigram)
        → MLP (多层感知机)
            → Activations & BatchNorm (训练稳定性)
                → Backprop Ninja (手动求导)
                    → WaveNet (CNN架构)
```

→ Wiki 概念页：反向传播与自动微分、多层感知机、批归一化、Softmax与交叉熵、权重初始化、字符级语言模型、WaveNet

### 第三层：现代架构 — "GPT 怎么造"

```
Let's build GPT (Transformer from scratch, 724万播放)
    → GPT Tokenizer (BPE分词)
        → Reproduce GPT-2 (124M参数完整复现, 4h)
```

→ Wiki 概念页：GPT、Transformer、Self-Attention、Byte-Pair-Encoding、GPT-2

### 第四层：实践哲学 — "怎么用/怎么想"

| 素材 | 核心输出 |
|------|----------|
| Sequoia Ascent 2026 演讲 | Agentic Engineering、Vibe Coding、Jagged Intelligence |
| How I use LLMs | LLM 工作流、工具链集成 |

→ Wiki 概念页：Vibe Coding、Agentic Engineering、Software 3.0、Verifiability (AI)

### 第五层：历史语境 — "从哪来"

| 素材 | 核心输出 |
|------|----------|
| 33 years ago (LeCun 1989) | 深度学习技术演进的时间尺度 |
| RNN Effectiveness | 循环神经网络时代的遗产 |
| Recipe for Training NNs | 经典最佳实践 |
| microgpt (200行) | "GPT 可以被一个人理解" 的终极证明 |

### 层间关联

五层之间形成三条张力线，正是 Karpathy 思想的核心：

```
理论 ←──────────────────────────→ 实践
  ↑                                ↑
  概念框架 (Layer 1)              代码复现 (Layer 2, 3)
  ↑                                ↑
  历史语境 (Layer 5)              应用哲学 (Layer 4)

Fundamentals ←───────→ Agentic Era
  ↑                      ↑
  从零构建 (2018-2022)    AI 写代码 (2026)
  手动反向传播             代码萎缩
  micrograd → GPT-2       Agent 编排世界
```

## 主线三：方法论升级 — wiki 从仓库到平台

### 变化前 vs 变化后

| 维度 | 之前 | 之后 |
|------|------|------|
| 采集管道 | 1条（B站 MCP） | 3条（B站/YouTube/文章） |
| 原始档案类型 | 单一（B站视频） | 三种（视频+视频+文字） |
| 脚本工具 | 1个 (bilibili-save.py) | 3个（+ youtube-save.py + article-save.py） |
| MCP 管理 | 全开 | 选择性启动（默认只开 scrapling） |
| 部署调度 | UTC 04:00（实际 14:00+） | UTC 02:00（期望 12:00） |

### 关键发现

**1. 脚本 > MCP 的采集场景**
MCP 适合交互式查询（单次问答），但批量采集、长时间下载场景下独立脚本更可靠。当前分工：MCP 用于交互，脚本用于采集。

**2. 并行 subagent 模式可扩展**
今天用 4 个 subagent 同时摄入 16 项素材，每项 3-5 分钟，总耗时仅 ~5 分钟。模式可复用于任何素材批次。

**3. 课程体系 vs 零散知识**
Karpathy 的案例证明，当摄入的素材属于同一个课程体系时，wiki 页面的内在关联性远高于零散摄取。概念页之间形成引用链，而非孤岛。

### 当前瓶颈

- **Worker 索引**: 632 页 / 785.9 KB，距 1MB 限额还有约 200KB
- **调度不可靠**: GH Actions `schedule` 触发零次，依赖手动触发
- **搜索覆盖**: 400+ 实体/概念已远超最初设计规模，关键词搜索可能开始退化

---

## 核心结论

1. **采集管道三合一是基础设施层面的质变** — wiki 不再局限于特定内容平台，任何有价值的信息源都可接入
2. **Karpathy 五层知识体系是全量摄取的案例验证** — 16项素材覆盖从概念到代码的全谱系，展示了 "一个人可以被完整吸收进知识库" 的可能性
3. **wiki 的下一步瓶颈不在采集而在检索** — 400+ 实体 + 600+ 页索引开始逼近搜索限额，需要思考 Worker 搜索的分层策略
